## Start Lab

This challenge is based on the Diffie-Hellman Key Exchange (DHKE) protocol.
In a standard Diffie-Hellman exchange, two parties (Alice and Bob) agree on a shared secret without ever sending the secret over the network. However, in this challenge, one of the private keys has been leaked.


#### The Math Behind It

In Diffie-Hellman, the math works like this:
- ``p`` = A large prime number (public)
- ``g`` = A primitive root modulo ``p`` (public)
- ``a`` = Alice's private key (secret)
- ``b`` = Bob's private key (secret)
- ``A = g^a mod p`` = Alice's public key
- ``B = g^b mod p`` = Bob's public key
The Shared Secret:
Both parties calculate the same shared secret S:
Alice calculates: ``S = B^a mod p``
Bob calculates: ``S = A^b mod p``.





##### The Exploit

Since the challenge leaked one private key (let's say Alice's ``a``), and we have Bob's public key B, we can just calculate:
``Shared Secret = pow(B, a, p)``

#### Analyze the Encryption Logic

Look at this specific line in ``encryption.py``:
````
enc = bytes([x ^ (shared % 256) for x in flag])
````

This means:
- It calculates the ``shared`` secret.
- It takes the last byte of that secret (``shared % 256``) to create a single-byte XOR key.
- It XORs every byte of the flag with that key.


To reverse it, we just need to calculate the same ``shared`` secret, get the same XOR key, and XOR the encrypted message again! (Because in XOR math, ``A ^ B ^ B = A``).


#### Identify the Leaked Values
Look at ``message.txt``. We have everything we need:
- ``A``: The Server's Public Key.
- ``b``: The Client's **LEAKED** Private Key.
- ``p``: The large prime number.
- ``enc``: The encrypted flag in hexadecimal.
